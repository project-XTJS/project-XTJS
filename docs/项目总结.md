# XTJS 智能审标平台 · 整体项目总结

> 本文按「**用过什么方法 → 为什么更换 → 现在效果**」梳理整个项目(前端 `XTJS-frontend` + 后端 `project-XTJS`)的每个业务点/技术点及其演进。
>
> 说明:其中近期(登录与四级权限、完整性并集、△ 加分项、语义判定、偏离/查重时机下放)为直接经手并验证的改动;更早的方法演进(OCR 升级、错别字模型更换、数据库重构、查重策略、模块化拆分等)依据各仓库提交历史与代码现状梳理,个别 commit hash / 行数为历史挖掘所得、非逐一手工复核。

## 一、项目定位与全景

**XTJS(信投建设智能审标平台)** 是一套招投标文件自动化审查系统,双仓库:

- **后端** `project-XTJS`:FastAPI,负责 OCR、8 大分析服务、数据/存储、认证。
- **前端** `XTJS-frontend`:React + Vite,负责项目管理、分析编排、结果审核(PDF 高亮 + 人工复核)。

核心工作流:**上传 → 分阶段 OCR(招标→商务标→技术标)→ 多维分析 → 结果审核/人工复核 → 导出报告**。全栈自托管(本地 GPU 模型,无外部云 API)。

## 二、技术栈总览

| 层 | 选型 |
|---|---|
| 后端框架 | FastAPI + Uvicorn,统一响应 `{code,message,rid,data}` |
| 数据库 | PostgreSQL,**psycopg2 原生 SQL + 连接池**(未用 ORM),Flyway 迁移,结果存 JSONB |
| 存储/缓存 | MinIO(预签名 URL)、Redis(应用缓存 + Celery) |
| AI 能力 | PaddleOCR-VL(OCR/版面/印章/签名)、BGE 向量(语义)、ERNIE-CSC/MacBERT(错别字) |
| 部署 | Dockerfile(CUDA GPU)+ docker-compose 全栈 + Makefile + Flyway 容器 |
| 前端 | React 18 + HashRouter + **Context(无 Redux)** + 原生 fetch 封装(两层缓存)+ 自定义 CSS(无组件库)+ Vite |

## 三、各业务点/技术点的「方法演进」(核心)

### 1. OCR 引擎
- **方法**:PaddleOCR 3.3 → 3.4 → **PaddleOCR-VL(视觉语言多模态)**;OCR 服务由 3000+ 行大文件重构为 Mixin 架构(engine / layout / signature)。
- **为什么换**:旧版对招标文件的复杂**表格、印章、签名、版面**识别力不足;VL 多模态显著提升结构化识别。
- **效果**:产出结构化 `layout_sections / logical_tables / seals / signatures` + bbox 坐标,支撑后续所有分析与前端高亮;GPU 加速、CPU 可降级。

### 2. 招标文件处理策略
- **方法**:原来"所有文件都跑 OCR、返回通用结果" → 改为**招标文件不强制 OCR,直接返回精确问题页码 / bbox / 行号**。
- **为什么换**:大文件 OCR 慢,且前端需要精确定位做高亮/导出。
- **效果**:延迟下降,前端高亮精度大幅提升(`location_utils` 做 bbox 合并)。

### 3. 错别字检测
- **方法**:词典/规则 →(一度引入 LanguageTool)→ **pycorrector + MacBERT / ERNIE-CSC 深度学习纠错**。
- **为什么换**:LanguageTool 依赖 Java、性能有限;神经纠错准确率更高,去掉 JRE 依赖、可 GPU 加速。
- **效果**:中文纠错准确率提升、镜像更轻;支持显隐开关 `TYPO_CHECK_VISIBLE`。

### 4. 完整性检查(integrity)
- **方法**:从招标"投标文件要求"提取必备清单 + 附件同义词库匹配;近期改为**「目录 ∪ 附件模板」并集去重,每项只算一次**。
- **为什么换**:单一来源会漏项(目录列了但附件没列,或反之)。
- **效果**:需求清单更全且不重复计数;响应偏离表也被纳入,不再漏检。

### 5. 一致性检查(consistency)
- **方法**:纯文本比对 → 加入 **BGE 向量语义匹配 + 结构化一致性(structured_consistency)+ 模板正文比对**;保留"模板正文过短(<20字)不做"规则。
- **为什么换**:纯字面比对误报高、识别不了语义等价;引入向量做标题/段落语义对齐(阈值 0.78 / 0.80 等)。
- **效果**:从"形式存在性检查"升级到"语义级一致性",误报下降。

### 6. 偏离表检查(deviation)★/△
- **方法演进**:基础表格+关键词 → ★ 强制项提取(章节范围识别 + 逐行聚合)+ **综合匹配算法**(SequenceMatcher 相似度 + 关键词片段命中 + 偏离模式正则)→ 近期**新增 △ 加分项 + BGE 语义符合度判定 + 时机下放到商务标 OCR 完成(status=2)**。
- **为什么换**:① 只识别 ★、缺 △ 加分项;② 命中后还需判断"是否大体符合";③ 原来卡在技术标完成才能跑,太晚。
- **效果**:★缺失=失败、△缺失=提示不失败;每个标记项全量"爆出来"带语义分(符合/存疑/待人工);商务标偏离/查重在 status=2 即可运行。

### 7. 查重(duplicate)
- **方法**:精确哈希匹配 + SequenceMatcher 相似度 → **技术标只查完全相同(去相似)**;商务标扩展范围 + 集成人员复用;新增**查重聚合(cluster)**与证据呈现。
- **为什么换**:技术标用相似度会大量虚报;商务标需要更全的范围与可读的聚类展示。
- **效果**:虚报下降、覆盖更全,前端能看懂重复关系。

### 8. 报价合理性 / 分项报价表
- **方法**:单个 2000~3500 行大文件 → 拆成模块(直接报价 / 下浮率 / 招标限价同义词 / 算术校验 `arithmetic_checker`)。
- **效果**:支持大小写金额一致性、下浮率合规、行/列/表三级加总校验;可维护性大幅提升。

### 9. 数据库设计
- **方法**:早期零散多表(含 `xtjs_file_content` 中间表)→ **规范化**:projects / documents / project_documents(多对多)/ result(JSONB)/ users。
- **为什么换**:冗余、查询低效;且要支持 `parsing_status` 分阶段状态机与后续权限表。
- **效果**:JSONB 灵活扩展分析结果、软删除、索引完善;迁移走 Flyway 版本化。

### 10. 异步与稳定性
- **方法**:同步阻塞 → **分阶段异步 + 并发限流 + 分批**(`parsing_status` 0/1/2/3 + 进度监控)。
- **为什么换**:大文件 OCR 易 OOM、长任务无反馈。
- **效果**:避免 OOM,支持大批量,前端可轮询进度。

### 11. 代码架构(工程方法)
- **方法**:多个 3000+ 行巨石文件 → **Mixin 多重继承 + FastAPI Depends 依赖注入 + 统一结果 Normalizer**;deviation / duplicate / unified / itemized / reasonableness 全部模块化拆分(30+ 文件)。
- **效果**:单一职责、易测试、可复用,为后续优化打基础。

### 12. 认证与权限(近期新增)
- **方法**:原来**零认证全公开** → JWT(HS256)+ bcrypt + **四级权限**(普通/中级/高级/管理员)+ 登录失败锁定 + 启动建初始管理员;前端 Context 鉴权 + 401 自动登出。
- **为什么换**:从内测走向实际部署,需要账号安全与权限隔离。
- **效果**:业务接口全局登录守卫,管理员可管账号;前端按 `hasLevel(n)` 显隐。

### 13. 前端展示与定位
- **方法**:简单结果列表 → **PDF 页码高亮定位**(收集 phrases/bbox/rects → `getDocumentPreview` 后端渲染高亮)+ **人工复核草稿编辑**(保存草稿 / 重算 / 确认)+ 结果规范化 `normalizeProjectResultsPayload`。
- **为什么换**:几百页 PDF 手工翻页找问题不可用;自动检查有误判需人工修正。
- **效果**:点问题项自动跳页高亮(最多 24 框);复核草稿后端持久化、可续编;后端返回格式升级时前端无感。

## 四、当前架构与设计模式

- **后端**:FastAPI + Depends 注入 + Mixin 组合 + 统一响应/异常 + 两层缓存 + 分阶段门控 `parsing_status`。
- **前端**:HashRouter + Context 鉴权 + `xtjsApi.js` 高内聚 HTTP 层(token 注入 / 401 登出 / 内存+session 两层缓存 / inflight 去重)。
- **AI 编排**:`UnifiedBusinessReviewService` 统一编排 8 大检查,结果经 Normalizer 标准化为 `{issues, summary, status}`。

### 8 大分析服务一览
| 服务 | 方法核心 | 关键文件 |
|---|---|---|
| 完整性 integrity | 需求并集 + 同义词匹配 + 计数去重 | `analysis/compliance/integrity.py` |
| 一致性 consistency | 模板正文比对 + BGE 语义 + 结构化 | `analysis/compliance/consistency.py` |
| 偏离表 deviation | ★/△ 提取 + 综合评分匹配 + BGE 语义 | `analysis/deviation/` |
| 报价合理性 reasonableness | 大小写金额/下浮率/限价 | `analysis/reasonableness/` |
| 分项报价 itemized | 表格重建 + 行/列/表算术校验 | `analysis/itemized/` |
| 签字盖章日期 verification | 印章/签名锚点 + 日期/截止比对 | `analysis/verification.py` |
| 查重 duplicate | 精确哈希 + 相似度 + 聚合 | `analysis/duplicate_check/` |
| 人员复用 personnel_reuse | 表格人员抽取 + 跨文档匹配 | `analysis/bid_document_review.py` |

## 五、现状效果与可改进点

**达到的效果**:全流程自动化审标可用、生产级 OCR、多维语义检查(向量 + 正则 + 模型)、安全认证、PDF 精确定位 + 人工复核闭环;后端单测通过、前端可构建。

**可改进点**:

2. 测试覆盖偏少(后端有单测,前端缺单测/E2E)。
3. 向量能力已统一到 BGE;原编排中未实际使用的 Milvus / etcd 已从 docker-compose 移除。
4. 文本清洗逻辑多处重复,可抽象复用。
