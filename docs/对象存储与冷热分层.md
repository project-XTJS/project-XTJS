# 对象存储布局、DB 瘦身与冷热分层

本文档说明本次升级中“数据库瘦身（大 JSON 外置 MinIO）+ MinIO 分级目录 + 对象生命周期/冷热分层”的落地方案、对象键约定、迁移与运维操作。

## 1. 背景与目标

OCR 识别内容 `xtjs_documents.content` 与项目分析结果 `xtjs_result.result` 原先直接以 JSONB 存 PostgreSQL，随项目增长线性膨胀。本次把这两类大 JSON 外置到 MinIO：数据库只保留元数据与对象键引用，库体积显著下降；同时把对象按“项目 / 角色 / 公司”分级组织，并对纯派生缓存配置自动过期。

## 2. 对象键布局（MinIO）

由 `MinioService.build_project_object_key()` 统一生成，桶为 `MINIO_BUCKET_NAME`（默认 `update-file`）：

| 类别 | 对象键 | 说明 |
|---|---|---|
| 招标文件（原件） | `<项目>/招标文件/<文件名>` | 法定留存 |
| 投标文件（原件） | `<项目>/投标文件/<公司>/{商务标\|技术标}_<文件名>` | 法定留存 |
| 识别内容（外置主存储） | `JSON识别/content/<文件名>_<doc-id>.json.gz` | gzip 压缩，DB content 置 NULL 后这里是唯一副本 |
| 人工复核工作副本（外置主存储） | `JSON识别/review/<文件名>_<doc-id>.json.gz` | gzip 压缩，DB review_content 置 NULL 后这里是唯一副本 |
| 分析结果（外置主存储） | `<项目>/JSON识别/result.json.gz` | gzip 压缩，DB result 置 NULL 后这里是唯一副本 |
| 预览图缓存（派生） | `cache/previews/...` | 可重算，短 TTL 过期 |

> 命名安全化：项目名/公司名/文件名会去除 `\\/:*?"<>|` 等非法字符并限长。

## 3. 数据库瘦身（读写路径）

- 写入：`create_document_with_content` / `update_document_content` 把识别内容写 MinIO，落 `xtjs_documents.content_object_key`，`content` 列置 NULL；`update_document_review_content` / `update_document_review_input` 把人工复核工作副本写 MinIO，落 `review_content_object_key`，`review_content` 列置 NULL；6 个结果写方法（`create_or_replace_project_result`、`upsert_project_result_item`、`update_project_manual_review_result`、`update_project_manual_review_workflow_scope`、`clear_project_manual_review_latest_result` 等）把完整结果写 MinIO，落 `xtjs_result.result_object_key` + 轻量 `result_keys`（顶层键列表，供项目列表统计），`result` 列置 NULL。
- 读取：统一经 `app/service/document_blob_store.py` 访问器；**优先用对象键 → MinIO**，缺键时回退数据库内联 JSON（兼容尚未迁移的历史行）。集中 hydrate 点：`get_document_by_identifier`（content + review_content）、`get_project_documents_for_duplicate_check`（content + review_content，effective_content 提取改在 Python 侧）、两处人工复核 FOR UPDATE SELECT、`find_reusable_ocr_document`（content）；`_sanitize_project_result_record`（result）。
- 事务安全：MinIO 写在 SQL 之前，写失败抛异常 → 事务回滚，不会留下指向坏对象的行。

### 表结构变更（Flyway `V20260627120000__add_blob_object_keys.sql`）

```sql
ALTER TABLE xtjs_documents ADD COLUMN IF NOT EXISTS content_object_key        TEXT;
ALTER TABLE xtjs_documents ADD COLUMN IF NOT EXISTS review_content_object_key TEXT;
ALTER TABLE xtjs_documents ALTER COLUMN review_content DROP NOT NULL;
ALTER TABLE xtjs_result    ADD COLUMN IF NOT EXISTS result_object_key TEXT;
ALTER TABLE xtjs_result    ADD COLUMN IF NOT EXISTS result_keys JSONB NOT NULL DEFAULT '[]'::jsonb;
ALTER TABLE xtjs_result    ALTER COLUMN result DROP NOT NULL;
```

## 4. 存量迁移

```bash
python tools/migrate_blobs_to_minio.py --dry-run          # 先统计待迁移行数
python tools/migrate_blobs_to_minio.py                    # 迁移 content / review_content / result（逐行提交，可重入）
python tools/migrate_blobs_to_minio.py --only reviews     # 仅迁移 review_content
python tools/migrate_blobs_to_minio.py --vacuum           # 迁移后 VACUUM (FULL) 回收空间
```

脚本把历史 `content` / `review_content` / `result` 写入 MinIO 分级键、回填对象键、JSONB 列置 NULL；已迁移行（对象键非空）自动跳过。`--only` 可选 `documents` / `reviews` / `results`。

## 5. 冷热分层与生命周期

| 前缀 | 性质 | 默认策略 |
|---|---|---|
| `cache/previews/`、`cache/` | 纯派生缓存，可重算 | 短 TTL 过期（默认 14 天） |
| 未完成分段上传 | 垃圾 | 7 天后中止清理 |
| `JSON识别/`（content/result） | **外置主存储** | **默认不过期**（过期=丢数据，需重新 OCR） |
| `<项目>/招标文件/`、`<项目>/投标文件/` | 原件，法定留存 | 长期保留；可选转冷层 |

```bash
python tools/setup_minio_lifecycle.py --show              # 查看当前规则
python tools/setup_minio_lifecycle.py --dry-run           # 预览将写入的规则
python tools/setup_minio_lifecycle.py --preview-ttl 14    # 应用：仅过期派生缓存
# 原件转冷层（需先配置远端冷层）
mc ilm tier add minio COLD --endpoint ... --access-key ... --secret-key ... --bucket ...
python tools/setup_minio_lifecycle.py --enable-cold --cold-tier COLD --originals-cold-days 90
```

> ⚠️ 注意 Feature 4 与 Feature 6 的关系：DB 瘦身后 `JSON识别/` 是 OCR 内容/结果的唯一副本，因此**默认不对其设过期**。仅当你接受“到期后需重新 OCR/重算”的语义时，才用 `--expire-json-recognition --json-ttl N` 显式开启。

## 6. 合规留存（可选）

对象锁定/保留期（Object Lock / Retention）需在**建桶时**开启版本控制与对象锁定，无法对既有桶直接追加。如需合规留存，建议新建带 Object Lock 的桶并迁移原件，再以保留期策略管理。本次默认仅做“原件长期保留 + 可选冷层转储”。
