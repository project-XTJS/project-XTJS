# 前端接口对接文档

服务地址：`http://127.0.0.1:8080`

## 1. 基础说明

- 除特殊说明外，接口统一返回：

```json
{
  "code": 200,
  "message": "success",
  "rid": "请求ID",
  "data": {}
}
```

- 分页参数统一优先用：
  - `page`
  - `page_size`
- 同时兼容旧参数：
  - `limit`
  - `offset`

- 文档类型：
  - `tender`：招标文件
  - `business_bid`：商务标
  - `technical_bid`：技术标

- 两个特殊接口：
  - `GET /api/postgresql/documents/{identifier_id}/source`：返回 `307` 跳转，不是普通 JSON
  - `GET /api/postgresql/documents/{identifier_id}/preview/pages/{page}`：成功时直接返回预览 JSON，不包 `code/message/data`

## 2. 项目接口

## 附录：统一分析接口

### 项目服务分析

`POST /api/analysis/run`

请求示例：

```json
{
  "project_identifier": "船体广告",
  "services": [
    "business_bid_format_review",
    "business_bid_duplicate_check",
    "technical_bid_duplicate_check",
    "personnel_reuse_check",
    "typo_check"
  ],
  "max_evidence_sections": 5,
  "max_pairs_per_type": 0
}
```

说明：

- `project_identifier`：项目标识
- `services`：可选服务列表，当前支持：
  - `business_bid_format_review`
  - `business_bid_duplicate_check`
  - `technical_bid_duplicate_check`
  - `personnel_reuse_check`
  - `typo_check`
- `consistency_check`、`verification_check` 已调整到 `task_type`，不再放在 `services` 中
- 返回结果为逐项执行结果，包含 `status / summary / items / results`

### 文本分析

`POST /api/analysis/run`

支持的 `task_type`：

- `integrity_check`
- `consistency_check`
- `verification_check`
- `pricing_reason`
- `itemized_pricing`
- `deviation_check`
- `full_analysis`

通用文本分析示例：

```json
{
  "task_type": "full_analysis",
  "text": "这里放待分析文本"
}
```

一致性检查示例：

```json
{
  "task_type": "consistency_check",
  "text": "{\"tender_document\": {\"data\": {}}, \"bid_document\": {\"data\": {}}}"
}
```

说明：

- `consistency_check` 的 `text` 必须传 JSON 字符串
- JSON 中需要同时提供模板文档和待检文档
- 支持的键名组合包括：
  - `model_json` + `test_json`
  - `model` + `test`
  - `tender_document` + `bid_document`
  - `tender_document` + `business_bid_document`

签字盖章日期检查示例：

```json
{
  "task_type": "verification_check",
  "text": "{\"tender_document\": {\"data\": {}}, \"bid_document\": {\"data\": {}}}"
}
```

纯文本兜底示例：

```json
{
  "task_type": "verification_check",
  "text": "这里直接放待检测的正文文本"
}
```

说明：

- `verification_check` 支持传 JSON 字符串或纯文本
- 传 JSON 时会优先做招标文件与投标文件的成对校验
- 传纯文本时只做单文档兜底扫描

## 8. OCR JSON 直传接口

如果前端拿到的不是原始 PDF，而是 OCR JSON，可以直接调这些接口：

- `POST /api/postgresql/projects/business-bid-format-review/upload-json`
- `POST /api/postgresql/projects/business-bid-duplicate-check/upload-json`
- `POST /api/postgresql/projects/technical-bid-duplicate-check/upload-json`
- `POST /api/postgresql/projects/personnel-reuse-check/upload-json`
- `POST /api/postgresql/projects/typo-check/upload-json`

这几类接口的共同特点：

- 表单上传 JSON 文件
- 会自动创建项目、文档、绑定关系
- 返回结构基本都是：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "project": {
      "identifier_id": "项目ID"
    },
    "result_key": "某个结果键",
    "result": {},
    "result_record": {},
    "document_binding": {}
  }

```

## 10. 前端推荐对接方式

### 10.1 页面建议

- 项目列表页：`GET /api/postgresql/projects`
- 文档列表页：`GET /api/postgresql/documents`
- 绑定管理页：`GET /api/postgresql/relations`
- 结果页：`GET /api/postgresql/projects/{identifier_id}/visualization-data`
- 原文预览弹窗：`GET /api/postgresql/documents/{identifier_id}/preview/pages/{page}`
- 打开源文件：`GET /api/postgresql/documents/{identifier_id}/source`

### 10.2 节省 OCR 成本的推荐流程

1. 创建项目
2. 上传招标文件
3. 上传商务标
4. 只绑定招标文件和商务标
5. 先做商务标形式审查
6. 再做商务标查重
7. 商务标不过则停止，不上传技术标
8. 商务标通过后再上传技术标
9. 更新绑定关系补上技术标
10. 最后做技术标查重

## 11. 最近接口调整

### 11.1 项目批量导入与两阶段 OCR

新增接口：
- `POST /api/postgresql/projects/batch/ingest-recognize`
- `POST /api/postgresql/projects/{identifier_id}/continue-technical-ocr`

当前推荐流程：
1. 一次上传项目名、招标文件、多组商务标和技术标。
2. 首次接口先处理：
   - 招标文件 OCR
   - 商务标 OCR
   - 商务标形式审查
   - 商务标查重
3. 技术标先上传并绑定，但默认不立即 OCR。
4. 业务人员确认后，再调用 `continue-technical-ocr` 继续技术标 OCR。
5. 技术标 OCR 完成后，再调用技术标查重、错别字、一人多用等接口。

### 11.2 `/api/analysis/run` 参数调整

现在 `/api/analysis/run` 除了原来的文本分析，还支持项目维度按服务执行分析。

项目服务模式的 `services` 现在支持：
- `business_bid_format_review`
- `business_bid_duplicate_check`
- `technical_bid_duplicate_check`
- `personnel_reuse_check`
- `typo_check`

文本分析模式的 `task_type` 现在支持：
- `integrity_check`
- `consistency_check`
- `verification_check`
- `pricing_reason`
- `itemized_pricing`
- `deviation_check`
- `full_analysis`

其中：
- `consistency_check` 已从 `services` 调整到 `task_type`
- `verification_check` 已从 `services` 调整到 `task_type`
- 这两个值不再作为项目服务枚举传入

示例：

```json
{
  "project_identifier": "船体广告",
  "services": [
    "business_bid_format_review",
    "business_bid_duplicate_check",
    "technical_bid_duplicate_check",
    "personnel_reuse_check",
    "typo_check"
  ]
}
```

### 11.3 按项目查结果接口默认改为轻量返回

以下接口默认不再返回过大的重复 JSON：
- `GET /api/postgresql/projects/{identifier_id}/results`
- `GET /api/postgresql/projects/{identifier_id}/results/{result_key}`
- `GET /api/postgresql/projects/{identifier_id}/merged-results`
- `GET /api/postgresql/projects/{identifier_id}/visualization-data`

默认返回重点：
- `project`
- `results`
- `available_result_keys`
- `result_record_meta`

按需展开参数：
- `include_raw_results=true`
  用于额外返回原始 `raw_results`
- `include_result_record=true`
  用于额外返回完整 `result_record`
- `view=raw`
  用于单项结果或项目结果接口返回原始结果视图

说明：
- 默认模式已针对前端展示做减重，响应体明显缩小。
- 如果前端只是展示结果页，优先直接用默认返回。
- 只有在调试或需要原始 pair 结果时，再显式加这些参数。

### 11.4 查重结果默认优先使用 merge 后结构

按项目查结果时，查重相关数据默认优先返回 merge 后的多文件聚合结构，而不是两两 pair 结构。

默认展示口径：
- `business_bid_duplicate_check`
- `technical_bid_duplicate_check`

如果前端需要查看原始两两对比结果：
- 调 `GET /api/postgresql/projects/{identifier_id}/results?include_raw_results=true`
- 或调单项接口并传 `?view=raw`

### 11.5 空项目和 upload-json 返回口径修正

已修正两类联调问题：

- `upload-json` 接口参数校验失败时：
  - 现在返回 `422`
  - 不再因为响应序列化问题误报 `500`

- 空项目执行商务标形式审查时：
  - 现在返回 `200`
  - 与其他分析接口保持一致
  - 返回中会标记为空结果

### 11.6 项目列表新增聚合字段

`GET /api/postgresql/projects` 现在额外返回这些聚合字段，前端可直接用于列表展示：

- `tender_count`
- `business_bid_count`
- `technical_bid_count`
- `document_count`
- `extracted_document_count`
- `pending_document_count`
- `result_available`
- `analysis_result_count`
- `available_result_keys`
